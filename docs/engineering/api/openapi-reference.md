## 1. API Overview

### Base URL

```
Production: https://api.secondbrain-os.com
Development: http://localhost:8000
```

### Authentication

All endpoints require authentication via one of:

1. **JWT Bearer Token** (Supabase Auth):
   ```
   Authorization: Bearer <supabase_access_token>
   ```
   Extracted and validated by `config.core.auth.get_current_user` dependency.

2. **API Key** (for automated clients):
   - Rotated via `POST /api/v1/auth/rotate-key`
   - Keys stored as Supabase user metadata

### Auth Flow

1. User signs in via Google OAuth (handled by Supabase Auth)
2. Frontend receives JWT session token
3. Every API call includes `Authorization: Bearer <token>`
4. Backend validates JWT, extracts `user_id`, filters all queries

### Versioning

- **URL-based:** All endpoints under `/api/v1/`
- **Deprecation headers:** `Deprecation: true` + `Sunset: <RFC1123 date>` on deprecated endpoints
- **Migration:** Breaking changes go under `/api/v2/`; v1 deprecated with min 6-month notice

### Pagination Convention

All list endpoints accept standard pagination query parameters:

| Parameter | Type | Default | Max | Description |
|---|---|---|---|---|
| `limit` | integer | 20 (varies) | 100-500 (varies) | Items per page |
| `offset` | integer | 0 | — | Number of items to skip |

Responses use one of two patterns:

**Paginated wrapper (tasks router):**
```json
{
  "data": [...],
  "total": 42,
  "limit": 20,
  "offset": 0
}
```

**Direct array (most routers):**
```json
[
  { "...resource fields..." }
]
```

### Standard Error Format

```json
{
  "detail": "Human-readable error message",
  "error_code": "TASK_NOT_FOUND",
  "request_id": "uuid-string",
  "timestamp": 1718000000.0
}
```

For Supabase errors, `detail` may contain the underlying Supabase error message.

### Common Status Codes

| Code | Meaning |
|---|---|
| 200 | Success (GET, PUT, POST with no `status_code` override) |
| 201 | Created (POST endpoints) |
| 202 | Accepted (async operations) |
| 204 | No Content (DELETE endpoints) |
| 400 | Bad Request — validation error or Supabase error |
| 401 | Unauthorized — missing/invalid auth token |
| 404 | Resource not found |
| 409 | Conflict — duplicate resource |
| 422 | Unprocessable Entity — Pydantic validation failure |
| 429 | Too Many Requests — rate limit exceeded |
| 500 | Internal Server Error |

### Rate Limiting

- **Global middleware:** 100 requests/minute per IP (configurable via `RATE_LIMIT_MAX`, `RATE_LIMIT_WINDOW`)
- **Per-endpoint:** Chat (`/api/v1/chat`) limited to 30 req/min via `EndpointRateLimiter`
- **Automation (`/api/v1/automation`):** Rate-limited via `EndpointRateLimiter`
- **Headers returned:** `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`
- See [rate-limiting.md](./rate-limiting.md) for full details.

### Request ID

Every request receives a unique `X-Request-ID` header (UUID v4). It can be provided by the client, or auto-generated by the server. All log entries include this ID for tracing.

### 4.1 Tasks (`/api/v1/tasks`)

**Schema module:** `database.schemas.task` — `TaskCreate`, `TaskUpdate`, `TaskResponse`, `TaskListResponse`

**Auth:** All endpoints require `get_current_user` dependency.

| Method | Path | Summary | Pagination |
|---|---|---|---|
| GET | `/api/v1/tasks/` | List all tasks | ✅ (limit, offset) |
| POST | `/api/v1/tasks/` | Create a new task | — |
| GET | `/api/v1/tasks/{task_id}` | Get a task by ID | — |
| PUT | `/api/v1/tasks/{task_id}` | Update a task | — |
| DELETE | `/api/v1/tasks/{task_id}` | Delete a task | — |
| POST | `/api/v1/tasks/{task_id}/complete` | Mark a task as complete | — |

**GET `/` Query Params:**
| Param | Type | Default | Description |
|---|---|---|---|
| `limit` | int | 20 | Items per page (1-100) |
| `offset` | int | 0 | Pagination offset |

**Response (GET /):** `TaskListResponse`
```json
{
  "data": [
    {
      "id": "uuid",
      "user_id": "uuid",
      "title": "string",
      "status": "pending|in_progress|completed|cancelled|missed",
      "priority": "low|medium|high|urgent",
      "due_date": "ISO datetime|null",
      "created_at": "ISO datetime",
      "updated_at": "ISO datetime",
      "completed_at": "ISO datetime|null",
      "estimated_minutes": "int|null",
      "category": "string|null",
      "description": "string|null",
      "project_id": "uuid|null",
      "goal_id": "uuid|null",
      "is_recurring": "boolean|null",
      "recurring_frequency": "string|null",
      "dependency_id": "uuid|null",
      "missed_count": "int|null"
    }
  ],
  "total": 42,
  "limit": 20,
  "offset": 0
}
```

**POST `/` Request Body (TaskCreate):**
- `title` (string, required, 1-500 chars)
- `description` (string, optional)
- `priority` (enum: `low`, `medium`, `high`, `urgent`, default: `medium`)
- `category` (string, optional)
- `estimated_minutes` (int, optional)
- `due_date` (ISO datetime, optional)
- `scheduled_start` (ISO datetime, optional)
- `goal_id` (uuid, optional)
- `project_id` (uuid, optional)
- `is_recurring` (boolean, optional)
- `recurring_frequency` (string, optional)
- `dependency_id` (uuid, optional)
- Default status: `pending`

**PUT `/{task_id}` Request Body (TaskUpdate):** Partial update. All fields optional.

**POST `/{task_id}/complete`:** Sets `status=completed`, `completed_at=now`. No request body.

**Error Codes:**
| Code | Status | When |
|---|---|---|
| 400 | Bad Request | Supabase insert/update/delete error |
| 404 | Not Found | Task ID does not exist for this user |

### 4.5 Chat (`/api/v1/chat`)

**Schema module:** `database.schemas.chat` — `ChatRequest`, `ChatResponse`

**Rate Limit:** 30 requests/minute per IP (enforced by `EndpointRateLimiter`)

| Method | Path | Summary | Pagination |
|---|---|---|---|
| GET | `/api/v1/chat/` | List chat conversations | — |
| POST | `/api/v1/chat/` | Send a chat message to ARIA | — |

**GET `/`:** Returns list of conversation objects, each with `id`, `title`, `lastMessage`, `timestamp`, `messageCount`. Grouped by `conversation_id`.

**POST `/` Request Body (ChatRequest):**
```json
{
  "message": "string (required)",
  "conversation_id": "string (optional)",
  "context": "string (optional, system prompt override)"
}
```

**Query Params:**
| Param | Type | Default | Description |
|---|---|---|---|
| `stream` | bool | false | Enable SSE streaming response |

**Response (ChatResponse):**
```json
{
  "response": "string",
  "action_taken": "string|null"
}
```

**Streaming:** When `stream=true`, returns `text/event-stream` with SSE events:
```
data: {"token": "partial text..."}
data: {"token": "more text..."}
data: {"done": true, "full_response": "complete text"}
```

**Fallback:** If LLM is unavailable, uses keyword-based fallback routing (checks for "task", "goal", "course", "habit", "help" keywords).

**Context Assembly:** The endpoint queries the user's pending tasks, active goals, courses, habits, sleep logs, time entries, and recent chat history to build context for the AI. Memory summary is fetched via `memory_agent.get_memory_summary`.

  "error_code": "ERROR_CODE",
  "request_id": "uuid-string",
  "timestamp": 1718000000.0
}
```

The `error_code` field is not consistently populated across all routers. Some routers may include it, while others return only `detail`.

### 5.2 Paginated Response (TaskListResponse pattern)

Used by the tasks router. Other routers return flat arrays.

  "data": [ { "...resource..." } ],
  "total": 42,
  "limit": 20,
  "offset": 0
}
```

### 5.4 Orchestrator Response (AI-powered endpoints)

## 6. Rate Limiting

| Scope | Limit | Window | Enforced By |
|---|---|---|---|
| Global (all endpoints) | 100 requests | 60 seconds | `RateLimiter` middleware |
| Chat (`/api/v1/chat`) | 30 requests | 60 seconds | `EndpointRateLimiter` |
| Automation (`/api/v1/automation`) | Uses `EndpointRateLimiter` check | 60 seconds | `EndpointRateLimiter` |

**Headers returned on all responses:**
- `X-RateLimit-Limit`: Maximum requests per window
- `X-RateLimit-Remaining`: Requests remaining in current window
- `X-RateLimit-Reset`: Unix timestamp when the window resets

**429 Response:**
```json
{
  "detail": "Rate limit exceeded. Max 100 requests per 60s"
}
```

See [rate-limiting.md](./rate-limiting.md) for full details.

## 7. Authentication

### JWT Bearer Token

The `get_current_user` dependency (from `config.core.auth`) performs:
1. Extracts `Authorization: Bearer <token>` header
2. Validates JWT using configured `JWT_SECRET` and `JWT_ALGORITHM`
3. Returns user object with `user.id` and other Supabase user data
4. All database queries are filtered by this user ID

### API Key Auth

API keys can be used as an alternative to JWT. Keys are:
- Generated and stored via Supabase user metadata
- Rotated via `POST /api/v1/auth/rotate-key`
- Validated per-request

### Refresh Token

JWT tokens can be refreshed via `POST /api/v1/auth/refresh` with a `refresh_token` in the request body.

### Row-Level Security (RLS)

All Supabase tables have RLS enabled. The backend enforces an additional layer by explicitly filtering all queries with `.eq("user_id", current_user.user.id)`.
