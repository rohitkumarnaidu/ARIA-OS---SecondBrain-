## 1. Executive Summary

The reporting system in Second Brain OS generates periodic summaries of user activity, system performance, and AI effectiveness. Reports serve dual purposes: they provide the user with productivity insights (daily briefings, weekly reviews) and provide the developer with operational visibility (monthly performance, quarterly business reviews). Reports are generated by scheduled cron jobs (for user-facing reports) and on-demand API calls (for operational reports). Distribution happens via the in-app notification system and the briefing/review API endpoints.

---

## 2. Purpose

Reports translate raw data into actionable summaries. For the user, they provide reflection and motivation (daily progress, weekly achievements). For the developer, they provide operational insight (system uptime, error trends, resource usage). Automated report generation ensures consistency and regularity without manual effort.

---

## 3. Scope

This document covers:

- Daily summary report (user's daily briefing)
- Weekly review report (user's weekly reflection)
- Monthly performance report (system metrics)
- Quarterly business review (product and business health)
- Report generation pipeline and scheduling
- Distribution channels (in-app, email)
- Report templates and metric inclusions
- Custom report creation
- Report archiving and retention

Out of scope: external BI tools (Tableau, Power BI), PDF report generation, scheduled email delivery.

---

## 4. Business Context

Reports are a core value proposition of Second Brain OS: the AI-powered daily briefing and weekly review are among the most used features. They provide reflection and planning support that a traditional to-do list cannot. On the operational side, monthly and quarterly reports help the developer track system health and make data-driven product decisions.

## 5. Functional Specification

### 5.1 Report Types

| Report | Frequency | Audience | Generated By | Distribution |
|---|---|---|---|---|
| Daily Briefing | Daily (7 AM) | User | `briefing_agent` | In-app notification + API |
| Weekly Review | Weekly (Sun 8 PM) | User | `weekly_review_agent` | In-app notification + API |
| Monthly Performance | Monthly (1st) | Developer | Scheduler cron | Console + monitoring API |
| Quarterly Business Review | Quarterly | Developer | Scheduler cron | Console + monitoring API |

### 5.2 Daily Briefing

Generated by the Briefing Agent (`packages/ai/agents/briefing_agent.py`) using the `prompts/agents/briefing_agent.md` prompt.

**Included Metrics:**
- Tasks due today (count)
- Tasks overdue (count)
- Habits to complete (list)
- Courses with upcoming deadlines
- Sleep quality summary (if sleep data available)
- Yesterday's completion rate
- AI-generated motivation message
- Top 3 recommended priorities

**Generation Pipeline:**
1. 7:00 AM cron trigger
2. Fetch user data (tasks, habits, courses, sleep logs)
3. Invoke `briefing_agent` LLM call (Ollama with Claude fallback)
4. Store result in `daily_briefings` table
5. Send in-app notification

### 5.3 Weekly Review

Generated by the Weekly Review Agent (`packages/ai/agents/weekly_review_agent.py`).

**Included Metrics:**
- Tasks completed this week (count + % of total)
- Habits tracked (streak summary)
- Courses progressed
- Goals progressed
- Income summary (if tracked)
- Time distribution (tasks by category)
- AI-generated insights and recommendations
- Goals for next week

**Generation Pipeline:**
1. Sunday 8:00 PM cron trigger
2. Aggregate weekly data from all modules
3. Invoke `weekly_review_agent` LLM call
4. Store result in `weekly_reviews` table
5. Send in-app notification

### 5.4 Monthly Performance Report (Operational)

Generated for the developer to review system health.

**Included Metrics:**
- API request volume (total, per endpoint)
- p50/p95/p99 latency
- Error rate breakdown (4xx, 5xx)
- AI token usage (total, per agent)
- AI provider distribution (Ollama vs Claude)
- Circuit breaker events
- Scheduler job success rate
- Active user count
- Storage usage
- Monthly cost estimate (AI API calls if Claude used)

### 5.5 Quarterly Business Review (Operational)

Generated for strategic product decisions.

**Included Metrics:**
- All monthly performance metrics (aggregated)
- User growth / retention trends
- Module adoption trends
- Feature flag impact analysis
- Technical debt metrics
- Security posture summary
- Roadmap progress (% complete against planned)
- Cost trends and projections
- Recommendations for next quarter

## 9. Data Models

### 9.1 Daily Briefing Schema

```python
class DailyBriefing(BaseModel):
    id: str
    user_id: str
    date: date
    generated_at: datetime
    tasks_due: int
    tasks_overdue: int
    habits_to_complete: list[str]
    courses_with_deadlines: list[dict]
    sleep_summary: Optional[dict]
    completion_rate_yesterday: float
    motivation_message: str
    recommended_priorities: list[str]
    ai_generated: bool  # whether AI was used or algorithmic fallback
    provider: Optional[str]  # ollama, claude, or algorithmic
    raw_response: Optional[str]  # LLM raw output
```

### 9.2 Weekly Review Schema

```python
class WeeklyReview(BaseModel):
    id: str
    user_id: str
    week_start: date
    week_end: date
    generated_at: datetime
    tasks_completed: int
    tasks_completion_rate: float
    habits_tracked: int
    longest_streak: int
    courses_progressed: list[dict]
    goals_progressed: list[dict]
    income_summary: Optional[dict]
    time_distribution: dict[str, float]
    insights: list[str]
    recommendations: list[str]
    next_week_goals: list[str]
```

### 9.3 Operational Report Schema

```python
class OperationalReport(BaseModel):
    id: str

    period_start: date
    period_end: date
    generated_at: datetime

    latency_p50: float
    latency_p95: float
    latency_p99: float

    error_rate_5xx: float
    ai_token_usage: dict[str, int]
    ai_provider_distribution: dict[str, float]
    circuit_breaker_events: int
    scheduler_success_rate: float
    active_users: int
    storage_usage_mb: float
    estimated_cost: float
    recommendations: list[str]
```

## 11. Security

- Reports contain user-identifiable data; served only to the authenticated user
- Operational reports are accessible only with admin privileges
- AI-generated content is stored as raw text; sanitised for PII before storage
- Report data retention is 12 months; older reports are archived and deleted
- LLM raw responses stored for debugging; excluded from report API responses

## 13. Edge Cases

| Edge Case | Handling |
|---|---|
| No data available for period | Report generated with empty sections and "no data" message |
| AI provider unavailable for briefing | Algorithmic fallback: compute metrics without LLM text |
| Multiple briefings in one day (rerun) | Overwrite existing briefing for same date |
| User has not completed onboarding | Briefing includes getting-started tips instead of metrics |
| Report generation takes too long | Timeout after 2x expected duration; return partial report |
| Historical report requested beyond retention | Return 404 with archival reference |

---

## 14. Failure Scenarios

| Scenario | Impact | Mitigation |
|---|---|---|
| Cron job misses trigger time | Report not generated until next cycle | Manual trigger available via API |
| AI agent returns malformed JSON | Report fields missing | Schema validation; fallback defaults |
| Database write fails after generation | Report generated but not stored | Cache in memory; retry write on next cron cycle |
| Report generation overlaps with another | Double data or conflict | Enforce lock: only one generation per report type at a time |

---

## 15. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Reports become stale (user ignores them) | Medium | Medium | Personalise content; vary format; A/B test |
| AI-generated insights are inaccurate | Medium | Low | Algorithmic metrics are always accurate; AI adds qualitative layer |
| Report storage grows unbounded | Low | Medium | 12-month retention; auto-archive older reports |
| Operational reports are not reviewed | Medium | Low | Auto-generate and store; review triggered by alerts |

## 18. Implementation Notes

### 18.1 Report Templates

Each report type has a template defined in `prompts/templates/`:

- `context_assembly.md` — Data assembly for AI agent inputs
- `email_templates.md` — Digest and notification templates

### 18.2 Adding a New Report Type

1. Define the report schema in `packages/database/schemas/`
2. Create the SQL aggregation query (if operational report)
3. Create the prompt template (if AI-generated report)
4. Create the generation function in `packages/ai/agents/` or `services/scheduler/crons/`
5. Register the cron job in `services/scheduler/main.py`
6. Add the report API endpoint in `apps/api/app/api/`
7. Add tests in `tests/test_scheduler.py` and `tests/test_api_endpoints.py`

### 18.3 Report Storage and Archival

```sql
-- Archive reports older than 12 months
INSERT INTO report_archive SELECT * FROM daily_briefings WHERE date < NOW() - INTERVAL '12 months';
DELETE FROM daily_briefings WHERE date < NOW() - INTERVAL '12 months';
```
